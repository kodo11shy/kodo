# 总览页改造方案：学生管理与收费

日期：2026-06-07

## 一、改造目标

本轮先改造两个模块：

```text
学生管理
收费管理
```

原则不是把列表搬到顶部，而是建立“老师一眼知道状态、马上能处理”的总览页。

成熟托育/家校产品的共同经验是：移动端管理页优先呈现待办、异常和今日/月度状态，而不是让老师从长列表里自己找问题。因此这两个页面都采用同一套信息架构：

```text
状态总览 -> 待处理队列 -> 筛选/搜索 -> 分组列表 -> 快捷动作
```

## 二、学生管理总览

### 页面定位

学生管理页不是通讯录，而是“班级主数据健康看板”。

老师进入页面后要立刻知道：

```text
班里现在有多少在读学生
今天谁来了
谁的家长还没绑定
谁的档案信息不完整
谁需要补接送/健康/联系方式资料
```

### 首屏结构

```text
┌────────────────────────────┐
│ 班级概况                    │
│ 在读 6   今日到班 0   家长绑定 0/11 │
│ 档案完整 4/6                │
└────────────────────────────┘

┌────────────────────────────┐
│ 待补资料                    │
│ 2 个孩子健康告知未确认       │
│ 3 个家长未绑定微信           │
│ 1 个孩子缺默认接送人         │
└────────────────────────────┘

搜索学生姓名、学校、班级

筛选：
全部 / 今日在托 / 未到 / 资料待补 / 家长未绑定

学生列表卡片...
```

### 顶部指标

| 指标 | 含义 | 当前接口可否支持 |
|---|---|---|
| 在读学生 | `students.status = 在读` | 已支持 `/students` |
| 今日到班 | 今日已签到人数 | 已支持 `/attendance/today` |
| 家长绑定 | 有 `ParentBinding.is_active` 的家长数 / 家长总数 | 需管理员接口或新增 summary |
| 档案完整 | 关键字段完整的学生数 / 在读学生数 | 前端可先根据详情计算，建议新增 summary |

### 学生卡片字段

每张学生卡保留轻量信息，但要比现在更有管理价值：

```text
头像字
姓名
年级 · 学校 · 班级
今日状态：已到 / 未到
资料状态：完整 / 待补
家长绑定：已绑定 / 未绑定
接送授权：已配置 / 未配置
```

卡片点击进入现有学生详情。

### 待补资料规则

MVP 先用可解释的固定规则：

```text
缺家长联系人：parents.length == 0
家长未绑定：家长无有效 ParentBinding
缺接送授权：pickups.length == 0
健康告知未签：health.consent_signed != true
基础资料待补：grade/school_name/school_class/address 任一为空
```

### 筛选设计

```text
全部
今日在托
未到
资料待补
家长未绑定
```

新开班状态下：

```text
今日在托 = 0
未到 = 全部学生
过程数据为空，但基础档案仍能展示
```

### 空状态

不显示“加载失败”式文案。

```text
还没有学生
请先录入学生和家长信息
```

如果有学生但没有签到：

```text
今天还没有签到记录
签到后会显示今日在托状态
```

### 实现策略

#### MVP 前端聚合

当前先不新增接口也能做：

```text
GET /students
GET /attendance/today
GET /students/{id} 逐个取详情
```

缺点：学生多时请求多，但当前托班约 20 人，可以接受。

#### 后续推荐接口

新增：

```text
GET /api/students/overview
```

返回：

```json
{
  "total": 6,
  "checked_in": 0,
  "parent_bound": 0,
  "parent_total": 11,
  "complete_count": 4,
  "issue_count": 3,
  "students": [
    {
      "id": 1,
      "name": "张小明",
      "grade": "大班",
      "school_name": "中心幼儿园",
      "school_class": "大（1）班",
      "attendance_status": "未到",
      "parent_bound": false,
      "profile_complete": true,
      "has_pickup": true,
      "health_signed": true,
      "issues": ["家长未绑定"]
    }
  ]
}
```

## 三、收费管理总览

### 页面定位

收费页不是财务系统，也不是付款工具，而是老师的小账本。

老师进入页面后要立刻知道：

```text
本月应收多少
已收多少
还有谁没交
哪些孩子只有部分缴费
最近记录了哪些账
```

### 首屏结构

```text
月份选择                     记一笔

┌────────────────────────────┐
│ 本月收费                    │
│ 应收 0      已收 0      未收 0 │
│ 已缴 0人    未缴 0人    部分 0人│
└────────────────────────────┘

┌────────────────────────────┐
│ 待处理                      │
│ 还没有生成本月应收账单       │
│ 或：3 个孩子未缴费           │
└────────────────────────────┘

筛选：
全部 / 未缴 / 部分缴 / 已缴

缴费学生卡片...
```

### 顶部指标

| 指标 | 含义 | 当前接口可否支持 |
|---|---|---|
| 应收总额 | 本月记录金额总和 | 已支持 `/payments/summary` |
| 已收 | 状态为已缴的金额 | 已支持 |
| 未收 | 未缴+部分缴金额 | 已支持，但部分缴更精细需增强 |
| 已缴人数 | 至少一条已缴记录且无未缴记录的学生数 | 前端可聚合 |
| 未缴人数 | 本月未缴或没有账单的学生数 | 需结合学生列表 |
| 部分缴人数 | 有部分缴记录的学生数 | 前端可聚合 |

### 缴费学生卡片字段

```text
学生姓名
本月状态：已缴 / 未缴 / 部分缴 / 未生成账单
本月金额：已缴金额 / 应收金额
费用项：托管费、餐费、材料费、其他
最近记录时间/备注
```

卡片点击后建议进入“该学生收费历史”，MVP 可先弹出明细或保留列表展开。

### 状态规则

```text
没有任何本月缴费记录 = 未生成账单
存在未缴记录 = 未缴
存在部分缴记录 = 部分缴
全部记录均为已缴 = 已缴
```

### 待处理队列

优先展示：

```text
未生成本月账单的学生
未缴学生
部分缴学生
金额为 0 或缺费用类型的异常记录
```

### 记一笔弹窗优化

当前弹窗可保留，但字段建议改名贴近用户：

```text
学生
费用项目
金额
所属月份
缴费状态
支付方式
备注
```

默认：

```text
所属月份 = 当前选择月份
状态 = 已缴
支付方式 = 微信转账
```

### 批量生成账单

建议作为 V2，不在第一轮强做。

入口可以预留为：

```text
生成本月应收
```

功能：

```text
按收费标准给所有在读学生生成托管费/餐费/材料费未缴记录
```

这会让收费总览真正有“应收”概念，否则现在只能统计已录入记录。

### 实现策略

#### MVP 前端聚合

当前可用：

```text
GET /students
GET /payments/summary?month=YYYY-MM
```

前端用学生列表和缴费明细合并，生成：

```text
studentPaymentRows
paymentStats
todoRows
```

#### 后续推荐接口

新增：

```text
GET /api/payments/overview?month=YYYY-MM
POST /api/payments/generate-monthly
```

`overview` 返回每个学生的月度账单状态；`generate-monthly` 用于批量生成应收。

## 四、视觉与交互规范

### 共同风格

```text
背景：浅灰蓝工作台背景
顶部：一张总览卡，不做大面积渐变
状态：绿=完成，橙=待处理，红=风险，蓝=信息
按钮：主按钮只保留一个高优先级动作
列表：卡片密度适中，便于扫读
```

### 不做的事

```text
不做营销式大标题
不做大面积插画
不把页面塞成后台表格
不让老师进入三层菜单才能处理问题
```

## 六、验收标准

### 新开班状态

```text
学生管理：
- 能看到 6 个在读学生
- 今日到班为 0
- 未到列表显示全部学生
- 过程数据为空时页面不报错

收费管理：
- 本月应收/已收/未收均为 0
- 提示还没有本月收费记录
- 记一笔可正常打开
```

### 有数据状态

```text
学生管理：
- 签到后今日在托人数变化
- 资料待补筛选能筛出问题学生

收费管理：
- 新增一笔后汇总金额变化
- 未缴/已缴/部分缴筛选正确
```
